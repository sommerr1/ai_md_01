# Release Changelog

## Цель

Быстро и аккуратно подготовить релиз:
- обновить `VERSION`;
- добавить новый верхний блок в `CHANGELOG.md` по фактическим изменениям;
- не включать шум (кэши, временные файлы, бинарные артефакты), если пользователь не просил.

## Workflow

1. Проверить состояние репозитория:
   - `git status --short --branch`
   - `git diff --stat`
   - `git log --oneline -10`
2. Прочитать текущие файлы:
   - `VERSION`
   - `CHANGELOG.md`
3. Определить тип bump через `AskQuestion`:
   - `patch` (`x.y.z -> x.y.(z+1)`)
   - `minor` (`x.y.z -> x.(y+1).0`)
   - `major` (`x.y.z -> (x+1).0.0`)
   - `custom` (пользователь вручную задает версию)
4. Собрать последние изменения из diff:
   - код: `src/**`, `tests/**`;
   - API/контракты: `docs/specs/**`;
   - эксплуатация: `docs/operations/**`;
   - сценарии/архитектура: `docs/scenarios/**`, `docs/architecture/**`.
5. Сформировать релизный блок в формате Keep a Changelog:
   - `## [<new_version>] - <YYYY-MM-DD>`
   - секции по необходимости: `Added`, `Changed`, `Fixed`, `Removed`, `Security`.
6. Вписать новый блок **сразу после вводной части** `CHANGELOG.md` (над предыдущим релизом).
7. Обновить `VERSION` до новой версии.
8. Показать пользователю краткий отчет:
   - старая/новая версия;
   - какие пункты добавлены в changelog;
   - какие файлы затронуты.
9. Коммит и push делать только если пользователь явно попросил.

## Правила качества

1. Писать changelog по факту изменений из git, а не по предположениям.
2. Формулировать пункты коротко: что изменилось и зачем.
3. Избегать технического шума (`__pycache__`, `.log`, `.sqlite3`, временные файлы), если пользователь отдельно не просит включать их в релиз.
4. Не дублировать старые пункты из предыдущих релизов.
5. Дата релиза берется текущая (локальная дата среды).

## Шаблон блока

```md
## [0.2.0] - 2026-06-05

### Added
- ...

### Changed
- ...

### Fixed
- ...
```

Секции без пунктов не добавлять.
